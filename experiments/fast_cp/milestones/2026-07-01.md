# 2026-07-01 Fast CP 里程碑

本文记录 2026-07-01 对 Hybrid Context Parallel / Fast CP 的目标整理、实现进展和实验结论。

## 今日定位

今天的工作把 fast CP 从“某个模型上的 CP 优化”推进成了一个更大的问题：

> 如何做一个高性能、可扩展、可迁移的序列并行库，并能根据 NVLink / PCIe / 跨节点等不同互联选择合适策略。

核心认知是：序列并行不是单纯换一个通信 primitive。它是一个调度问题，需要同时考虑：

- 通信是否是瓶颈。
- 计算是否被切得太碎，导致 GEMM / SDPA 利用率下降。
- 每次切分节省了多少计算，又新增了多少通信。
- pack/unpack、stream/event、CPU launch、NCCL kernel 数量是否超过收益。
- CUDA Graph / torch.compile 是否真的减少开销，还是引入新的边界。
- 每个优化理论上最多能省多少时间。

## 已完成的工程里程碑

### 1. 统一 CP core 的雏形

已有两类核心入口：

- 通用 self-attention CP：面向 Wan / Flux-like DiT。
- image + full text CP：面向 Z-Image / Qwen-Image-like joint attention。

当前已经覆盖或原型化：

- AGKV
- Ulysses
- Ring
- Ring graph
- Ring + Ulysses / unified-style hybrid
- full text + sharded image joint attention

评价：**功能骨架已经出现，但不是成熟高性能库。**

### 2. Ulysses all-to-all baseline 收敛

实验比较了 list `all_to_all`、packed `all_to_all_single`、symmetric tensor
`all_to_all_single`、以及 symmetric-memory pipelined P2P hook。

结论：

- 在不改变 projection 输出 layout 和 attention 输入 layout 的前提下，`all_to_all_single`
  是当前最合理的默认方案。
- symmetric tensor 路径可用，但没有决定性收益。
- pipelined symmetric P2P hook 当前 kernel 太碎，barrier/spin/copy 过多，不适合迁移。
- 下一层优化不该继续换 all-to-all API，而应减少 pack/unpack。

关键方向：

```text
current:
  projection -> [B, S, H, D]
  pack copy  -> [rank, B, S, H/cp, D]
  all_to_all_single

target:
  projection -> [rank, B, S, H/cp, D]
  all_to_all_single
```

### 3. Async QKV projection 原型

实现/验证过的思路：

- AGKV：K projection 后发起 K all-gather，V/Q projection 尝试覆盖通信。
- Ulysses：K/V/Q projection 与 input all-to-all 交错。
- Z-Image：packed QKV fused path。
- Qwen-Image：full text + image fused path。
- Wan：通用 self-attention `attention_fused` path。

结论：

- correctness 方向成立。
- harness 中能看到真实 overlap。
- 真实 NVLink 模型收益不稳定，整体偏中性或负收益。
- 该路径应保留为实验开关，不能作为默认优化。

### 4. torch.compile / CUDA Graph 边界被明确

观察到：

- `torch.compile` 不能自然覆盖 NCCL、stream、event、`record_stream` 这些 CP comm 路径。
- 为了让模型 compile 成功，CP attention 需要成为 eager island。
- eager island 意味着 compile 只能优化 CP 外围或 tail 中的小操作，无法把通信路径整体融合。
- CUDA Graph 在 ring 这种静态重复结构上更有价值。
- Ulysses/all-to-all + async overlap 目前没有证明足够收益。

结论：**compile / graph 是模式相关优化，不是全局银弹。**

### 5. txt+img joint 范式被单独提出

Z-Image / Qwen-Image 这类模型中，text token 和 image token 的并行策略不同：

- text token 通常较短，可以复制。
- image token 很长，需要切分。
- non-causal attention 沿 query 维可拆，但拆分 SDPA 会导致 kernel 变小、利用率下降。

今天尝试了 naive text/image split：

- 语义正确。
- trace 中能看到 image all-to-all 和 text SDPA overlap。
- 但一个大 SDPA 被拆成两个或多个小 SDPA 后，总 SDPA 时间增加，抵消了通信隐藏收益。

结论：**txt+img joint 是重要范式，但 naive query split 不值得迁移。**

## 今日关键数据

### Harness: async overlap

H20/NVLink 单元测试中：

| case | 结论 |
| --- | --- |
| AGKV eager overlap | on-node 快链路基本不赚 |
| AGKV 人为 comm-bound | overlap 可到约 5%-9% |
| AGKV graph overlap | harness 中可看到正收益 |
| Ulysses input overlap | 小幅改善或中性 |
| Ulysses `all_to_all_single` | 比 list all-to-all 更干净，kernel 数更少 |
| text/image split | 正确但变慢 |

代表性数据：

| 实验 | 结果 |
| --- | ---: |
| AGKV cp4, img=4096, H=24, eager overlap | -3.7% |
| Ulysses cp4, img=4096, H=24, eager overlap | -0.7% |
| AGKV cp4, img=8192, H=48, comm-bound overlap | +9.1% |
| packed `all_to_all_single` vs list all-to-all | Ulysses 绝对时间下降，trace kernel 数减少 |
| naive txt/image split | `txt_q/txt_out/txt_both` 均不优于 baseline |

### 真实模型: Z-Image / Qwen-Image

无 CUDA Graph、CP attention eager 的真实模型短步测试：

| 模型 | 配置 | overlap 结果 |
| --- | --- | ---: |
| Z-Image | cp2 | 约 +2% |
| Z-Image | cp4 | 基本持平或略慢 |
| Qwen-Image | cp2 | 约 +1%-2% |
| Qwen-Image | cp4 | 约 -2% |

结论：真实模型上没有稳定收益，尤其 cp4 下不应默认开启。

### 真实模型: Wan2.1-1.3B

cp4 Ulysses，打开/关闭 overlap 后复测 3-step，排除 step0 lazy init 干扰。

steady-state 结果：

| 配置 | steady forward avg |
| --- | ---: |
| overlap off | 约 468.95 ms |
| overlap on | 约 460.65 ms |

收益约 **1.8%**。

同时确认：

- 早先看到 CFG branch 时间差很大，主要来自 step0 初始化/预热污染。
- steady-state 下 CFG 两个 branch 时间基本均衡。
- 因此不能用 step0 数据判断 `cfp=2` 会产生严重等待。

## 今日最重要的负结论

1. NVLink 上 async overlap 不是主线收益点。
2. all-to-all API 已经基本收敛到 `all_to_all_single`，继续换 API 的空间很小。
3. `torch.compile` 无法直接消掉 CP comm 路径的复杂性。
4. naive txt/image split 会降低 SDPA 效率。
5. 真实模型短步测试中，async QKV overlap 的收益基本在噪声级别。

这些负结论很重要：它们说明 fast CP 的后续重点不应该是“到处 overlap”，而应该是：

- 减少通信字节。
- 减少 pack/unpack。
- 保持大 GEMM / 大 SDPA 的效率。
- 只在适合的模式上使用 CUDA Graph。
- 根据 topology 和 profile 做 runtime policy。

## 当前实现完成度

| 方向 | 完成度 | 说明 |
| --- | ---: | --- |
| 多 CP 策略原型 | 中 | AGKV/Ring/Ulysses/Unified/Ring graph 路径存在 |
| async QKV projection | 中 | 多模型已接，但性能未证明 |
| txt+img joint 抽象 | 中高 | 方向正确，应继续沉淀 |
| compile/graph 一键化 | 低到中 | ring graph 更可信，compile 仍需 eager island |
| 任意 DiT 适配 | 中 | Wan/Z/Qwen 已有路径，Flux 等仍需验证 |
| 高性能库成熟度 | 低 | 还处在实验平台和策略收敛阶段 |

一句话评估：

> 今天做出了 fast CP 的实验骨架和一些重要边界结论，但还没有做出足以称为高性能库的稳定性能收益。

## 下一步建议

### 近期

1. 把 async overlap 默认关掉，只保留 flag。
2. 把 `all_to_all_single` 作为 Ulysses 默认路径。
3. 清理 CP API，让普通 self-attention 和 full-text image attention 两类入口稳定下来。
4. 为每个 CP mode 建立固定 benchmark matrix：
   - harness microbench
   - trace
   - component profile
   - Z/Qwen/Wan 真实模型短步
   - 长步 sanity

### 中期

1. projection 直出 rank-major layout，减少 Ulysses pack。
2. ring CUDA Graph 工程化。
3. txt+img joint 的更聪明调度，避免拆小 SDPA。
4. PCIe 上继续推进 fp8 K/V、KV cache、贯穿式序列切分。
5. 建立 topology-aware policy：
   - NVLink：优先保持计算效率，谨慎 overlap。
   - PCIe：优先减少字节和消息碎片。
   - 跨节点：重新评估 AGKV overlap 和压缩通信。

### 长期

目标是让 ChituDiffusion 的 DiT 模型只暴露必要的 Q/K/V producer、RoPE、mask、txt/img
layout 信息，由 fast CP runtime 决定：

- 用哪种 CP mode。
- 是否 graph。
- 是否 compile。
- 是否 overlap。
- 是否压缩 K/V。
- 是否采用 hybrid topology 切分。

最终希望得到的不是某个模型的特化 trick，而是一套可解释、可测量、可迁移的序列并行调度系统。
